842. 排列数字

对于此题，首先可以在心中想象一棵排列树：

从此处便可知道，dfs 的搜索方式是“一条路走到黑”，先上 1，发现 2 还有，就上 2，发现 3 还没用，就上 3，最后上完了发现已经没有可用数字了，就回退到 “12_” 这一步，但是此时还是没有可用数字，就回退到 “1_ _” 这一步发现 3 还没用，就上 3，然后依次这样遍历下去。

在这个过程中，我们发现 dfs 的思考方式是这样的，如果走到实在无路可走，就回退到上一步，这就是 if 内的判断条件，如果能走，就往下走，记得进行标记和恢复现场这个两个过程。

845. 八数码

这题搜索思路其实不复杂，但是难就难在状态表达上面，每次都要改变字符串，这时我们不妨将 dis 用哈希来存，这样查找起来也更快。我的代码显得有些累赘，有很多 char 到 string 的转换，看了下 y 总写的，就非常清爽。

  • 每次找 x 的坐标可以使用string.find('x')找到字符 x 在队头的 string 中对应的下标，然后用int x = k / 3, y = k % 3;计算一下即可。
  • 这样交换的时候直接swap(t[a * 3 + b], t[k])，为了不改变原字符串的顺序，在 if 结束后，再 swap 回来就可以了。
  • 上面这样写就避免了来来回回的 char to string 的转换，省去了很多不必要的操作。

1113. 红与黑

这题的标签叫 flood fill ，从其大致意义上即可知道是要去填充东西，这题也是一样，对于这题来说，是要去“填充”所有能到达的黑色砖块，详细分析可见下面这段文字。

UNAC code

此题的错误写法。。。

这种写法目的实际上是，找到很多条路（没有路可以走的时候即为终点），算出每条路上所有的黑色砖块的个数，然后输出其最大值，但是这样是错误的。

为什么呢？首先题目的意思是，除了不能到达的点，我们要将所有的黑色砖块都走一遍，除了红色砖块，实际上不存在”没有路可以走“这种情况，因为这些砖块我们可以一直反复的走，但是这样达不到我们要的结果，我们的目的是要将所有能到达的黑色砖块都走一遍。所以，每个砖块仅仅代表一种状态只有”走“或”不走“两种情况，还有一点需要注意的是，如果能从起点走到一个黑色砖块 1 上，或者从起点走到黑色砖块 2 上，那么 2 一定可以走到 1，（这一点非常好想，假设如果 2 不能走到 1 ，那么 1 一定红色砖块或者被红色砖块包围的部分，而 1 可以从起点走到，所以 2 一定可以到达 1。）所以，这样搜索的方案就变成了如何走遍所有能走的砖块。

所以说，如果用 bfs， 就不需要回溯（恢复现场）的过程，此时的 dfs 计算的实际上是从某个点出发能到达的砖块数，将已经到达的砖块标记一下，之后的遍历就不再需要访问这个砖块了。

P1135 奇怪的电梯

一道比较简单的宽搜题，dis 状态记录的重点在于从某层楼到某层楼需要按几次按钮。且是由队头节点次数加一拓展而来。

P2895 [USACO08FEB]Meteor Shower S

这个题怎么说，想到了就很好做，但是想不到就会一直卡（sad）。开始没有对流星预处理，采取边搜边看的策略，bfs 答案不对，又换 dfs 写（），结果显然是更不对的。由于最后可能发生流星打不到人的状态，所以某个中间状态就已经安全了。而 边搜边看会不会被打到 就变得很不合理。

那么如何预处理呢？

  • 一是需要记录每个位置被流星打到的最小时间，这样才能知道某个时间某个位置能不能走。
  • 二是 dis 数组表示到某个点的时间，这样时间和空间就关联起来了。
